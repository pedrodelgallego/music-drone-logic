Replicate: Kaitlyn Aurelia Smith — Live Buchla

Copy the mechanism — a patch running semi-autonomously while the performer rides only a few controls — not the Buchla tone. Build a study where a self-running patch carries the music and your hands ride the edges, performed in one take with Live 12, Digitakt II, MicroFreak, and the MOTU M4, and A/B it against the reference at matched loudness.

The point

This drill teaches one thing: how to perform a system live — to set up a self-patching system (sequencer, random, and modulation feeding each other) generates the music; the performer intervenes only at a few points rather than playing it all, then ride it through a single recorded take. You are not arranging; you are operating. The reference gives you the clue; the take proves whether you understood it.

Before you start
  • Live 12 set saved as REP-KAITLYN-AURELIA-SMITH-LIVE-BUC.
  • Digitakt II connected by USB for sync and by main outs to MOTU M4 inputs 3/4.
  • MicroFreak output connected to MOTU M4 input 1 when the lesson needs a synth source.
  • MOTU M4 headphones at the Lab 001 mark; Live tracks set to Monitor Off while recording hardware.
  • Reference on a Live audio track named REF, with Utility inserted for loudness matching.
  • A panic path written on paper before any sound: mute → limiter → bypass → stop → fallback. On stage there is no second take.
  • What you will make: a 3–6 minute printed performance where the hands, the ride, and the system’s own behaviour define the form — not an edit.
  • What you will turn off: unrehearsed moves, unsafe paths, and any automation or edit added after the take.
  • What you will prove: the printed take contains the intended moves, performed live, without later repair.
  • What counts as success: remaining differences are source choices or rig limits, not a missing mechanism.
Reference pass · 10 minutes

Listen before you build

Listen to (and watch) one clear section of a Kaitlyn Aurelia Smith live Buchla performance. The Buchla Music Easel is patched so that its sequencer, random source, and envelopes feed each other — the system produces evolving melody and rhythm largely on its own. Smith performs by riding a handful of controls: timbre, the sequencer’s range, a filter, the balance between voices. Most of the notes are not chosen; they emerge from the patch. The skill is designing a system worth listening to without you, then knowing the few places where a hand makes it sing. Write one short answer for each question before you touch the gear.

  1. What is the smallest repeated behaviour you can point to?
    Answer

    Do not name the mood first. Name the behaviour: a self-patching system (sequencer, random, and modulation feeding each other) generates the music; the performer intervenes only at a few points rather than playing it all. The whole study hangs on isolating that one operating principle.

  2. Which device or path could create that behaviour in your studio?
    Answer

    Pick one place: Digitakt II pattern data, MicroFreak source/modulation, a Live device chain, the MOTU M4 input path, or a performance move. If the mechanism lives everywhere, you have not isolated it yet.

  3. What should stay boring so the mechanism remains audible?
    Answer

    Keep the supporting material plain. A good replication drill often has boring notes, boring drums, or boring source audio so the real mechanism — here, the live operation of the system — can be heard clearly.

  4. What would prove you performed the system rather than copied the style?
    Answer

    Your take should use different material while producing the same kind of behaviour, and it should be one continuous performance. If it only works when you copy the sound, or only after editing, the mechanism is still missing.

Steps

  • Button map for the steps below: use these controls whenever the lesson names the device.
  • Live 12: ⌘ T creates an audio track, ⇧⌘ T creates a MIDI track, ⌘ R renames the selected track, ⌘ , opens Settings, ⌘ M enters MIDI Map mode, and ⌘ J consolidates the selected clip.
  • Digitakt II: [TRK] + [TRIG 1-16] selects the active track. Press [RECORD] for grid recording, press numbered [TRIG] keys to place or remove hits, and press [PLAY] to hear the pattern.
  • Digitakt II sound and timing: press [SRC] for sample/source settings, [FLTR] for filter, [AMP] for envelope/level, [LFO] for modulation, [FX] for sends, and [TEMPO] for tempo and swing.
  • MicroFreak: use the Preset encoder to choose a plain starting sound, Type to choose the oscillator, Cutoff and Resonance for the filter, the cycling envelope and LFO for self-running modulation, Matrix for routing, and Save only after the study behaves correctly.
  • MOTU M4: MicroFreak usually enters input 1. Digitakt II main L/R enter inputs 3/4. Inputs 3/4 have no front-panel gain knobs, so set their record level from the Digitakt II Main Volume.
  • Performance controller: map your ride controls (FaderFox EC4, or any MIDI knobs) with ⌘ M before the take so every move is reachable without looking.
  • Sound target: use simple source material so the performance behaviour is exposed.
  • Recording target: peaks around -12 dBFS in Live before any processing.
  • Copy target: a solo flip between REF and your print makes the shared behaviour obvious.
  • Performance rule: one take, no repair. Change one variable per pass and write it down.
Phase 1 · ~20 min

Put the reference in Live and write the hypothesis

Do this
  1. Drop the reference onto a track named REF and loop the clearest 30–60 seconds.
  2. Add Utility to REF and lower it until it is comfortable.
  3. Write: “The mechanism is ___.” Use the listening tag as a first guess: a patch running semi-autonomously while the performer rides a few controls.
  4. Write the one thing you will not copy: sound, melody, sample, room, mix polish, or performance style.
  5. Create tracks named SYSTEM, RIDE, and PRINT.
  6. Write the panic path in order and tape it to the desk: mute, limiter, bypass, stop, fallback.
Expected result

You have a looped reference, named tracks, one written hypothesis, and a panic path — before any programming starts.

Phase 2 · ~35 min

Build the system and map the ride

The mechanism is a generative patch that would keep playing if you walked away, plus a small set of intervention points. Build the self-running system first, confirm it is musical with no hands on it, then perform by riding only the edges.

Do this
  1. Build a self-running patch: a MicroFreak arpeggiator/sequence modulated by its own LFO and cycling envelope, or a Live MIDI clip driven by Random/Scale devices — set it so it evolves with no input from you.
  2. Confirm autonomy: let it run for two minutes untouched and verify it is already worth listening to. If it needs your hands to be interesting, the patch is not finished.
  3. Then expose only three intervention points — timbre, sequence range/transpose, and a filter — and perform the take by riding those edges, leaving the system to generate the substance.
  4. Limit the whole performance to six controls maximum: two levels, two filters/effects or rule values, one mute, and one emergency stop or bypass.
  5. Set the limiter and emergency mute first if the system can run away. Then rehearse the first minute twice without recording.
Expected result

A working system that produces the target behaviour with your hands on only a few controls, and a rehearsed ride you can perform in one pass.

If it fails

If you find yourself triggering every event by hand, the system is not autonomous enough — push more of the motion into the patch, the rule, or the feedback, and pull your hands back to the edges.

Phase 3 · ~15 min

Route the studio and perform one full take

Make the signal path boring and dependable, then commit. The take is the deliverable; do not repair it afterwards.

MicroFreak · input 1 + Digitakt II · inputs 3/4 → MOTU M4 → USB → Live 12 · Monitor Off
Do this
  1. Plug sources in, set hardware tracks to Monitor Off, and check peaks near -12 dBFS — not “as loud as possible.”
  2. Confirm the limiter and emergency mute are in reach before you start the system.
  3. Start the system, count yourself in, and perform the whole arc in one pass — ride the controls, let the system breathe, recover live from any wobble.
  4. Do not punch in, overdraw automation, or edit after the take. If it fails, perform it again from the top.
  5. Press ⌘ J to consolidate the best full take into a clean clip named PRINT-A.
  6. Mark the three moments where your hands changed the form, and the one moment you recovered from a wobble.
Expected result

One continuous printed take where the form came from the system and your live ride, not from editing.

Phase 4 · ~20 min

Make the system map and the A/B proof

Draw the mechanism, then test the printed take against the reference at matched loudness. The map is how you find what failed.

Do this
  1. Put REF and PRINT-A on adjacent tracks and match loudness with Utility.
  2. Listen to 10 seconds of REF, then 10 seconds of PRINT-A.
  3. Write one sentence: “The first thing that gives mine away is ___.”
  4. Fix only that one thing in the system or the ride, then perform PRINT-B as a fresh full take.
  5. Keep both takes. The difference between A and B is the lesson.

Phase 5 · ~15 min

Diagnose one miss and write the rule

Do this
  1. Keep REF from Kaitlyn Aurelia Smith — Live Buchla one solo away and the current rig in place.
  2. Carry out a single corrective pass, changing only what proves a self-patching system (sequencer, random, and modulation feeding each other) generates the music; the performer intervenes only at a few points rather than playing it all.
  3. Record a short full proof pass before comparing — do not judge from a live monitor only.
  4. Write the one setting, patch change, or gesture that made the biggest difference.

Final check

Play the reference and your best take at matched loudness, then answer these in the logbook:

  • What mechanism did you perform?
    Answer

    Name the live operating principle, not the genre: a self-patching system (sequencer, random, and modulation feeding each other) generates the music; the performer intervenes only at a few points rather than playing it all.

  • Where does that mechanism live in your setup?
    Answer

    Point to the device or path: Digitakt II pattern, MicroFreak patch, Live device chain, MOTU M4 routing, or your hands during the take.

  • What did you deliberately not copy?
    Answer

    The answer should be material: melody, sample, exact sound, mix polish, or artist style. The workshop copies systems, not identity.

  • What rule carries into the related lab?
    Answer

    Write it as a usable instruction. Example: “If the take needs editing to work, I built an arrangement, not a performable system.”

  • Logbook — the miss list, three lines: Matched: ___ · Missed: ___ (best guess why) · Change next: ___
  • System diff sentence: “I thought the system worked like ___; after performing it, I discovered it worked like ___.”
  • Carry into the lab: open lab-027.html and reuse the corrected rule, not the reference sound.
