Write the Lab Note

The last étude turns you into the workshop. You write a complete lab page for your 034 system — specs, bench calibration, numbered build steps, a diagnosis table, a listening list — in this site's own format, as if a stranger were about to take your seat. The piece proved you can build it; this page proves you understand it, because the success test is brutal and simple: someone who has never met you rebuilds the system from the page alone.

Start here
  • Hard prerequisite: lab 034's full report bundle — decision log, patch notes, print map — filed and complete. This étude is written from those documents, not from memory.
  • Run the ten references below first — each is a piece of documentation that works; the question shifts one degree: what is the constraint, where is it audible — and how did the document carry it to a stranger?
  • Reread any one early lab page from Block 1 before writing — that page taught you a system you had never seen; yours must do the same job in reverse.
  • This is an S4 étude: no template rows are filled for you. The format is the site's; every word is yours. Expected time: one half-cycle of writing, then the cold-read test.
The point

Teaching is the final proof of understanding: what you can build you might have stumbled into, but what you can teach a stranger to build, you actually possess.

How this lab works

Transcribe the system

  • Work from 034's paper trail: decision log, patch notes, print map — if a step is not in the documents, you must rebuild that step to recover it.
  • Write both specs as the page's spec tables, exactly as the workshop's labs present them.
  • Every number a stranger needs — levels, tempos, settings, names — appears on the page, not in your head.

Author the page

  • The full anatomy, in the site's own format: the point · specs · bench preflight · numbered build steps · diagnosis table · listening list · pass gate.
  • The diagnosis table is symptom → cause → fix, seeded with your own 033 and 034 fault entries — your faults are the curriculum.
  • The listening list names the references your system steals from, each with one line on what the stranger should hear.

Run the stranger test

  • Give the page to someone who was not in the room — or to yourself after a real gap, read cold against the rig.
  • The reader attempts the rebuild from the page alone; every question they have to ask is a defect in the page, not in the reader.
  • This is R8's reproducibility test pointed forward: R8 asks if your past documentation can rebuild a piece; 035 writes documentation designed to pass.
Pass standard
Minimum passA complete lab page for the 034 system — specs, bench, numbered steps, diagnosis table, listening list — written in the site's format and filed.
Solid passThe page survives a cold read: a stranger (or you, after a gap) can rebuild the system from the page alone, and every question raised was logged and fixed.
Advanced transferThe page teaches the why as well as the how — each step names the rule it serves, so the reader could adapt the system, not just clone it.

New concepts — in plain words

ConceptIn plain words
The lab noteA teaching document in the workshop's own anatomy — the point, the specs, the bench, the steps, the diagnosis table, the listening. Not a description of what you did: an instruction set for what a stranger must do.
The stranger testThe only honest audit of documentation: someone with no access to your memory rebuilds from the page. Every "obviously you would…" that turns out not to be obvious is a hole, and the holes are invisible to the author by definition.
Curse of knowledgeOnce you know a system, you cannot remember not knowing it — the expert's steps silently skip what the expert no longer sees. The cure is mechanical, not moral: cold reads, real strangers, and a logged question list.
Diagnosis tableSymptom → cause → fix, in rows. The most valuable part of any manual and the part almost nobody writes, because it requires admitting what went wrong. Your three years of fault entries are exactly this table, waiting to be transcribed.
ReproducibilityThe archive's pass/fail: can the work be rebuilt from what was filed? A system that lives only in your hands dies with your attention; a system that lives on a page can be taught, repaired, and outlive the project file.

Listening — the hypothesis source

Ten works where the documentation is the study object — books, films, lectures, decks, and liner notes that carried a system from one mind to others. The Block 8 question still applies — what is the constraint, and where is it audible? — plus this étude's clause: how did the document transmit it?

01 Sisters with Transistors (2020) the archive decides who is heard

Lisa Rovner's documentary on electronic music's women pioneers — Oram, Derbyshire, Ciani, Spiegel, Radigue and others — built almost entirely from archival footage and the artists' own recorded voices. The film's lesson is this étude's premise inverted: these systems survived because tape, film, and paper were kept; history hears whoever filed their work.

Listen for
  • The primary sources: how much of the film is the artists demonstrating their own systems on camera, decades ago — documentation made while working, not after.
  • The gaps: where the film must rely on others' descriptions because no recording of the artist explaining herself survives.
  • Which systems you could begin to rebuild from the footage alone — and what the footage shows that prose could not.
After — verify

Whose voice does the archive carry best, and why? The artists who recorded themselves explaining their work — Ciani's demos, Spiegel's writings — arrive vivid and first-person; those who did not are narrated by others. That is the stake of this étude in one sentence: write your own page or someone else paraphrases you, badly or never. If the film read as history only, rewatch one studio demonstration and ask what a stranger could rebuild from it.

02 Brian Eno & Peter Schmidt — Oblique Strategies a decision system on cards

A deck of cards, one instruction each — "Honour thy error as a hidden intention", "Repetition is a form of change" — distilled from two working practices into the most compressed studio documentation ever printed. Fifty years on, strangers still run the system exactly as designed, which makes it the most reproducible document in this list.

Listen for
  • Study the deck (the full card list is easily found): each card is a complete, executable instruction with no dependency on the authors' presence.
  • The compression: every card began as a longer working note — what survived the cut, and what kind of sentence survives?
  • The protocol around the cards: drawn when stuck, trusted even when unwelcome — the usage rule is part of the documentation.
After — verify

Why do the cards still work on people the authors never met? Because each one documents a decision procedure, not a description — the reader executes it rather than learns about it. Audit your draft page against this standard: which of your build steps are executable, and which merely describe? Rewrite one descriptive step as a card-grade instruction. If the deck read as aphorisms, draw one card at your next stuck moment and obey it literally — the system only exists in use.

03 Tony Conrad — Completely in the Present (2016) what happens when you don't file

Tyler Hubby's documentary on a founding figure of drone minimalism whose central work went unreleased for decades — partly through dispute, partly through indifference to the archive — and very nearly vanished from the record. The cautionary tale of this list: a practice of enormous influence, reconstructed late and from the outside because the artist did not file it himself.

Listen for
  • The reconstruction effort: how much of the film is other people testifying to work they heard in rooms fifty years ago — memory standing in for missing tape.
  • Conrad's own ambivalence about documentation and legacy — a deliberate stance, with audible historical costs.
  • The moments where surviving recordings appear: what changes when the actual sound replaces the testimony about it.
After — verify

What did the missing archive cost? Decades of attribution, and a history written first by others. The étude's negative proof: your 034 system is currently in the Conrad state — alive in one person — and this page is what moves it out. Name the one part of your system that would be hardest to reconstruct from testimony alone; that part gets the most careful page section. If the film read as biography, rewatch ten minutes counting how often the evidence is a document versus a memory.

04 Brian Eno — A Year with Swollen Appendices the producer's logbook, published

Eno's 1995 diary published whole, with its "swollen appendices" — short essays on the working ideas the diary keeps touching. The structure is the lesson: dated, unglamorous daily entries (sessions, doubts, lunches) in front, and the distilled systems thinking filed behind. It is your logbook's published cousin, and the appendices are what your three years of entries are waiting to become.

Listen for
  • Read a week of entries: how studio decisions actually look at ground level — small, repetitive, undramatic — versus how retrospectives describe them.
  • The diary-to-appendix flow: find an idea that appears as a passing daily note, then again as a finished essay — that promotion is exactly this étude's move.
  • The candour as documentation policy: failures and boredom logged at the same resolution as breakthroughs.
After — verify

What makes this teaching rather than memoir? The appendices — the diary supplies evidence, the appendices extract the rules. Your lab page has the same anatomy: the 034 decision log is the diary; the page you are writing is the appendix. Pick one recurring note from your own logbook and promote it to a paragraph a stranger could use. The listening link is YouTube-only by nature — interviews where Eno works through the book's ideas aloud.

05 Alvin Lucier — Music 109 teaching as plain speech

Lucier's Wesleyan lecture course, transcribed: a composer explaining the experimental canon — Cage, Reich, Young, his own I Am Sitting in a Room, which you rebuilt in Block 5 — in sentences a first-year student can execute. No jargon, no theory armour; each piece described by what was physically done, in what order, and why.

Listen for
  • Read his chapter on a work you already know from the workshop and compare his explanation to your own worksheet — what does he include that you skipped?
  • The procedural spine: every description is essentially a numbered build — equipment, action, duration, what to listen for.
  • The trust in plainness: the most radical pieces get the simplest sentences, and lose nothing.
After — verify

Why does plain speech work for material this strange? Because the systems were always simple — a tape loop, a room, a sine wave — and plain words match the actual mechanics where theory would obscure them. The standard for your page: if a sentence in your draft needs its own explanation, it is two systems wearing one sentence, as lab 032 taught. Rewrite your most jargon-dense step the way Lucier would say it. Then play the piece his chapter described and check the words against the audio.

06 Daphne Oram — An Individual Note the inventor writes the manual

The 1972 book by the co-founder of the BBC Radiophonic Workshop and inventor of Oramics, the machine that turned drawn lines into sound. Part physics primer, part philosophy, part studio handbook, written in her own unmistakable voice — an inventor documenting not just how her system works but how she thinks, so the reader inherits both.

Listen for
  • Read a chapter, then listen to her Oramics work: the book's drawn-sound idea audible as glissandi and envelopes no keyboard would produce.
  • The dual register: rigorous signal explanation and personal speculation interleaved without apology — the voice survives the technical content.
  • Her questions to the reader — the book teaches by handing the inquiry over, not by closing it.
After — verify

What does the inventor's own manual carry that a third-party one cannot? The intent — why the machine exists, what musical question it answers — which is exactly what survives gear obsolescence. Oramics machines are museum pieces; the book still teaches. Your rig will date the same way; your page's "the point" section is what outlives it. If the book felt eccentric, reread one technical passage and notice it is also precisely correct — voice and rigour are not in competition.

07 Tony Visconti — RBMA lecture (Madrid 2011) one decision, perfectly told

The canonical entry from the Red Bull Music Academy lecture archive: Visconti narrating his production life, including the famous three-microphone gated corridor behind Bowie's "Heroes" vocal — a single engineering decision explained so completely (distances, gates, the singer's behavior) that it became reproducible folklore. Strangers have rebuilt that vocal chain from this telling alone.

Listen for
  • The "Heroes" passage: every element a rebuilder needs is spoken — mic positions, gate logic, why the singer must push louder to open the far mics.
  • The structure of a good war story: symptom (a dead-sounding hall), system (the gated mics), proof (the recording) — diagnosis-table thinking, told aloud.
  • What the lecture format adds over a manual: the failures, the room politics, the reasons — context no spec sheet carries.
After — verify

Why is this one story retold by engineers forty years on? Because it documents a complete causal chain — problem, mechanism, audible result — at rebuildable resolution. That is the bar for your page's build steps: each one should survive being retold by someone else without losing the numbers. Pick the one decision in your 034 build most worth a story and write it at Visconti resolution. Then play "Heroes" and hear the third microphone open on the last verse.

08 William Basinski — The Disintegration Loops liner notes the story filed with the audio

The documentation that travels with the record: Basinski's account of the decaying tape loops, the transfer that destroyed them as it preserved them, and the September 2001 rooftop where the music found its meaning. You rebuilt the audio system in Block 4's lesson; this time the study object is the note itself — a few hundred words that permanently changed how the audio is heard.

Listen for
  • Read the note first, then play five minutes: mark exactly which sounds the text taught you to interpret — the dropouts as decay, the hiss as time.
  • The technical disclosure inside the story: tape, loop, transfer, deterioration — the whole system is in there, rebuildable, wrapped in narrative.
  • Then the control test: imagine the audio with no note at all — what survives, and what becomes mere texture?
After — verify

Is the note part of the work? Functionally yes — the system's meaning is split between the audio and the document, and the workshop's print maps make the same claim at bench scale. The boundary lesson: the note tells the truth about the mechanism, which is why it amplifies the audio instead of decorating it; a fictional backstory would curdle on relisten. Your page's "the point" section makes this same pact — say what the system is, and let the meaning compound. If you doubt the note's power, play the piece for someone twice: once cold, once told.

09 KiNK — gear-walkthrough videos the rig explained by its owner

The performer whose live sets you studied in Block 7's Sofia and Valencia lessons, in his other role: sitting with his machines on camera, demonstrating exactly how each piece of the rig produces the moves the crowds saw — routing, settings, hand positions, mistakes included. Self-documentation by a working artist, made while the system is alive.

Listen for
  • The demonstration grammar: he shows the sound, then the cause, then the variation — symptom, mechanism, range, like a diagnosis table performed.
  • What video documents that prose cannot: hand speed, gesture size, which controls are touched together — the Block 7 layer of your own system that your page must capture in words or images.
  • The generosity calculus: he gives away the whole method and loses nothing, because the system without his hands and taste is only a parts list.
After — verify

Compare the walkthroughs to the Boiler Room sets you replicated: could you have rebuilt his moves from the performance footage alone, without the explanations? Mostly not — performance shows that, documentation shows how. Decide which parts of your 034 system need the video treatment (a photographed knob position, a drawn gesture) because prose would lose them. And note his lesson against secrecy: published methods built his audience, not his competition.

10 Pauline Oliveros — Sonic Meditations instructions a stranger can run

Twenty-five text scores from 1971 — "Take a walk at night. Walk so silently that the bottoms of your feet become ears." — each a complete musical system in a few sentences, executable by anyone, trained or not. You met her sounding work in Block 6; these pages are the purest pass of the stranger test in existence: no notation, no equipment, no author present, and the system runs.

Listen for
  • Run one meditation yourself, fully — the document only reveals its precision in execution.
  • The wording: every sentence is an instruction or a condition; nothing describes, nothing decorates — count the verbs.
  • What is deliberately unspecified — duration, place, who — and how the piece survives, even needs, that openness.
After — verify

How can four sentences be a complete system? Because they specify the invariants — attention, action, condition — and free everything else, which is exactly what your contract does and what your page must do: fix what matters, release what does not. A page that over-specifies is as unteachable as one that under-specifies; Oliveros marks the line. After running one score, audit your draft's steps: which constraints are load-bearing, and which are just how you happened to do it? Cut the second kind or mark them optional.

3 · Build

Build the lab in five passes. Each pass produces an artifact, not just a better-sounding session: a written rule, a printed comparison, a fault entry, and a corrected system sentence.

Build recipe · Write the Lab Note

Make the authored system audible

Do this in order
  1. Write the system sentence. Before building, write: this piece works by ___, while forbidding ___. This sentence is the brief.
  2. Author both specs. Fill the System Spec and Engineering Spec from a blank page. No hidden decisions; every path and boundary is written.
  3. Build the proof. Use Live, Digitakt II, MicroFreak, and MOTU M4 only where they prove the sentence. Remove anything decorative.
  4. Print and document. Render the proof and file the print map: audio, project state, device memory, written rule set, and what is gone forever.
  5. Run the stranger test. Try to rebuild from the page without memory. Every question is a defect in the documentation and gets fixed in the page.

The lab can move to diagnosis only when the printed audio proves the written rule. If the rule sounds true only while you explain it, simplify the system and print again.

Pass gate — étude complete when
  • The complete lab page exists — point, specs, bench, numbered steps, diagnosis table, listening list, pass gate — in the site's format, filed with the 034 bundle
  • The stranger test was run by the protocol, and the question log exists
  • Every logged question produced a fix on the page — defects patched in the document, not excused in person
  • The one prescribed-fault rule — what the deleted-obvious-step stall taught
  • One sentence of reflection: what teaching the system revealed that building it did not

Étude principle

The workshop's last skill is the workshop itself: a voice you can design, build, perform, and explain is yours for life — and the proof of all four is a page from which a stranger can rebuild it.
