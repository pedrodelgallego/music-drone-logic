Shared workshop references

Written once, used by every lab. These pages are stubs in the scaffold; the assessment frame below is already final.

The two specs

Every track lab and capstone opens with both, before any gear is touched. Year 1: provided, you verify. Year 3: you author them blank.

System Spec — composition-facing

  • Inputs — what raw material enters the system
  • Transforms — what changes it, in what order
  • Feedback / variation — what keeps it alive
  • Constraints — what prevents clutter
  • Human role — performed, decided, or curated

Engineering Spec — path-facing

  • Signal path · gain plan · clock and sync
  • Latency · stereo and phase · noise budget
  • Print strategy — what is committed, when, why
  • Failure mode — what will go wrong, how to diagnose it
  • Safety — feedback, headphone level, field, live

The fault ritual

Predict the symptom→ Create the fault→ Observe→ Diagnose the location→ Fix→ Write the rule

One controlled breakage per track lab, bench lab, and capstone — never punitive. Fault classes: gain, monitoring, clock, phase, routing, latency, noise, state, performance.

The lab report

Filed for every track lab, étude, and capstone
  • Rendered piece — 24-bit WAV, −3 dBFS
  • Routing diagram — drawn before patching, corrected after; the diff is the lesson
  • Recording plan — what gets printed, when, at what stage
  • Patch notes — the settings that mattered
  • Session notes
  • Reflection answers, citing the lab’s listening assignment
  • One removal decision — what was cut, and why
  • Fault entry — symptom, cause, fix, prevention rule
  • Print map — what exists as audio, MIDI, automation, patch state, memory
  • System diff — “I thought it worked like X; I discovered it worked like Y.”

The seven layers — the graduate frame

Used at capstone reviews, L8, and the final defense: describe any system through material, signal, control, time, decision, human, and archive layers.

Practising without the gear

Tool fallbacks

Every Year 2–3 practice has a free software equivalent, so no lab is gated by hardware. Modular patching — done in VCV Rack when no Eurorack is on the bench. Generative and rule-based systems — built in Pure Data, SuperCollider, or TidalCycles instead of a hardware sequencer or self-playing patch. Microtonal tuning — authored in Scala and loaded into any plugin or soft-synth that reads its scale files. No-input mixer feedback — modelled with a software feedback loop (a send routed back to its own input) inside the DAW or VCV Rack, with the same safety checklist as the hardware lab. The system is the lesson; the substrate is interchangeable.

Checking your own work

Self-validation aids

The solo learner has no instructor to confirm a result, so the workshop builds in external checks. Each lab's listening assignment supplies reference recordings to compare against — if your result sits far from every reference, the gap is a finding. The two specs arrive as example specs in Year 1 (S1–S2), so you can see the expected granularity before authoring your own. When a fault appears, run the comparison checks: is this fault real, or am I hearing my own monitoring? (verify on a second system and in mono) and is my gain right? (read the meter, don't trust the volume). Measure before you conclude — the metering toolkit above exists so "it sounds wrong" can become "it measures wrong here."

Going deeper on one voice

  • Optional lineage track: alongside the block programmes, follow one artist or tradition across roughly 30 years — early work, the shift, the late style. Hearing a single voice evolve teaches how a system matures into a signature, which the block-by-block survey can't show. Choose a lineage that crosses several blocks (a dub engineer, a modular composer, a techno producer, a regional drumming tradition) and revisit it at each year synthesis; it becomes a model for L8's "voice evolving" session and the final defense.
