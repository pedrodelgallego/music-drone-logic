Replicate: Paul Lansky — “Six Fantasies on a Poem by Thomas Campion”

Copy the reference’s working mechanism, not its style. Build a short original study where you analyse the pitch and rhythm of a spoken recording and let those measured contours — not notes you invent — drive a synthesised melody.

The point

This drill teaches one thing: a decision system — rule, boundary, state, seed, chance, or curation. The reference gives you the clue; the build proves whether you understood it. You are copying the system that makes the music, not the music.

Before you start
  • Live 12 set saved as REP-LANSKY-SIX-FANTASIES.
  • Digitakt II connected by USB for sync and by main outs to MOTU M4 inputs 3/4.
  • MicroFreak output connected to MOTU M4 input 1 when the lesson needs a synth source.
  • MOTU M4 headphones at the Lab 001 mark; Live tracks set to Monitor Off while recording hardware.
  • Reference on a Live audio track named REF, with Utility inserted for loudness matching.
  • What you will make: a 6–10 minute original system run where the rule generates material and you only curate the result.
  • What you will turn off: individual note-fixing, unlogged random choices, and rules without boundaries.
  • What you will prove: two runs behave like siblings — different output, same rule family.
  • What counts as success: remaining differences are source choices or rig limits, not a missing mechanism.
Reference pass · 10 minutes

Listen before you build

Listen to one clear stretch of Paul Lansky — “Six Fantasies on a Poem by Thomas Campion”. 1979: Lansky recorded a voice reading Campion’s poem, then used the computer to measure the speech — its pitch contour, its rhythm, its resonances — and resynthesised it six different ways, each fantasy emphasising a different feature the analysis exposed. The melodies you hear were not composed note by note; they are the melody already latent in the speaking voice, drawn out and amplified by rules that read the analysis data. The mechanism is: measurement first, then synthesis driven by the measurement. You will copy that pipeline at small scale — capture a contour, then let it play an instrument. Write one short answer for each question before you touch the gear.

  1. What is the rule, stated in one sentence?
    Answer

    Do not name the mood. State the procedure: here, a spoken voice analysed and resynthesised by computer rules into melody. If you cannot write it as an instruction someone else could follow, you have not isolated it yet.

  2. Which device or path could run that rule in your studio?
    Answer

    Pick one place: Digitakt II pattern data and trig conditions, MicroFreak source plus its Matrix, a Live MIDI/device chain (Random, LFO, Expression Control), or a written process score. If the rule lives everywhere, you have not isolated it yet.

  3. What should stay boring so the rule remains audible?
    Answer

    Keep the supporting material plain. A good replication drill often has boring notes, boring drums, or boring source audio so the real mechanism can be heard clearly.

  4. What would prove you copied the system rather than the style?
    Answer

    Your study should use different material while producing the same kind of behavior. If it only works when you copy the sound, melody, or sample, the mechanism is still missing.

Steps

  • Button map for the steps below: use these controls whenever the lesson names the device.
  • Live 12: ⌘ T creates an audio track, ⇧⌘ T creates a MIDI track, ⌘ R renames the selected track, ⌘ , opens Settings, ⌘ M enters MIDI Map mode, and ⌘ J consolidates the selected clip.
  • Digitakt II: [TRK] + [TRIG 1-16] selects the active track. Press [RECORD] for grid recording, press numbered [TRIG] keys to place or remove hits, and press [PLAY] to hear the pattern.
  • Digitakt II sound and timing: press [SRC] for sample/source settings, [FLTR] for filter, [AMP] for envelope/level, [LFO] for modulation, [FX] for sends, and [TEMPO] for tempo and swing.
  • Digitakt II sync: press [SETTINGS] for MIDI Config → Sync. Use [YES] and the arrow keys to turn clock and transport receive on or off.
  • MicroFreak: use the Preset encoder to choose a plain starting sound, Type to choose the oscillator, Cutoff and Resonance for the filter, Matrix for modulation routing, and Save only after the study behaves correctly.
  • MOTU M4: MicroFreak usually enters input 1. Digitakt II main L/R enter inputs 3/4. Inputs 3/4 have no front-panel gain knobs, so set their record level from the Digitakt II Main Volume.
  • Sound target: use simple source material so the decision behavior is exposed.
  • Recording target: peaks around -12 dBFS in Live before any processing.
  • Copy target: a solo flip between REF and your print makes the shared behavior obvious.
  • Variation rule: change one rule value per pass and write it down.
Your rig
Phase 1 · ~20 min

Put the reference in Live and write the rule

Do this
  1. Drop the reference onto a track named REF.
  2. Loop the clearest 30–60 seconds.
  3. Add Utility to REF and lower it until it is comfortable.
  4. Write the rule as one sentence: “The system a spoken voice analysed and resynthesised by computer rules into melody.” This sentence, not a sound, is what you are building.
  5. Write the one thing you will not copy: sound, melody, sample, room, mix polish, or performance style.
  6. Create tracks named SOURCE, SYSTEM, and PRINT.
  1. Work in Live 12 only for this phase: keep REF looped from Paul Lansky — “Six Fantasies on a Poem by Thomas Campion” and use MIDI effects, Random, LFO/Expression Control, and resampling.
  2. Build the rule — a spoken voice analysed and resynthesised by computer rules into melody — as the smallest Live set that proves it: a Random or Probability device inside a Scale boundary, an LFO mapped to one parameter, or a generator clip.
  3. Keep every source on named tracks, then use Utility to match the level against REF before judging it.
  4. Record or resample the result to a fresh audio track named PRINT; do not leave the proof only as devices and MIDI.
  5. A/B ten seconds of PRINT against REF and change only the rule value that most affects a spoken voice analysed, then resynthesised as melody.
  1. Use Digitone II for this phase: translate a spoken voice analysed and resynthesised by computer rules into melody into FM tracks, sound locks, trig conditions, LFOs, macros, and Overbridge or main-out recording.
  2. Program the pattern, source, or control path so it runs the rule itself, not just the surface of Paul Lansky — “Six Fantasies on a Poem by Thomas Campion”.
  3. Keep one clear performance control for the main change in this phase; name it before recording so the move is repeatable.
  4. Record the device into Live or a recorder as a full pass, then compare it with REF at matched level.
  5. If the target box suggests an easier shortcut, use it only when the same mechanism stays audible.
Expected result

You have a looped reference, three working tracks, and the rule written as one buildable sentence before any programming starts.

Phase 2 · ~40 min

Build the rule, not the notes — then let it run

This is the heart of the drill. Do not play a melody. Construct an analysis-to-control pipeline: record a short spoken phrase, extract its pitch contour and amplitude envelope, and route those measured curves to control a synth’s pitch and level — the voice composes, you only built the path. Keep the material plain so the rule, not the sound, is what you hear.

Do this
  1. Write three rows before sound: state (what the system remembers), seed (what makes a run repeatable), boundary (what it may never do). Leave no row blank.
  2. Build the rule itself: an analysis-to-control pipeline: record a short spoken phrase, extract its pitch contour and amplitude envelope, and route those measured curves to control a synth’s pitch and level — the voice composes, you only built the path. On Digitakt II this is trig conditions, probability locks, LFOs, and per-step parameter locks; on MicroFreak it is the Matrix and the sequencer.
  3. Choose one plain voice that can survive many repetitions. Avoid a flashy preset that sounds finished before the rule runs.
  4. Set the boundary as a hard limit (Scale device, min/max on a rack macro, a fixed key window) so the rule cannot escape it.
  5. Start a 10-minute unattended recording. Hands off unless the system clips or goes silent outside its written boundary.
  6. Record a second run with one rule value changed (a probability, a boundary, a seed). Do not edit individual notes between runs.
  7. Compare behavior across the two runs: event type, density over time, range, and forbidden events.

Phase 3 · ~15 min

Route the studio so the print is clean

Make the signal path boring and dependable. If the recording path is confusing, you will misdiagnose the mechanism.

MicroFreak · input 1 + Digitakt II · inputs 3/4 → MOTU M4 → USB → Live 12 · Monitor Off
Do this
  1. Plug MicroFreak into input 1 if the study uses it. Set the input gain while watching Live.

Draw the mechanism, then test the print against the reference. The map is not decoration; it is how you find what failed.

  4. Write one sentence: “The first thing that gives mine away is ___.”
  5. Change one rule value — not a note — and record PRINT-B.
  6. Keep both prints. The difference between A and B is the lesson.

Final check

Play the reference and your best print at matched loudness, then answer these in the logbook:

  • What mechanism did you copy?
    Answer

    Name the mechanism, not the genre: a probability rule, a selection table, a number formula, an analysis-to-control path, a stated melodic rule, a process score, a sequence mutation, or a behaving patch.

  • Where does that mechanism live in your setup?
    Answer

    Point to the device or path: Digitakt II pattern, MicroFreak patch, Live device chain, MOTU M4 routing, or the written score.

  • What did you deliberately not copy?
    Answer

    The answer should be material: melody, sample, exact sound, mix polish, or artist style. The workshop copies systems, not identity.

  • What rule carries into the related lab?
    Answer

    Write it as a usable instruction. Example: “If the behavior disappears when the source changes, I copied the sound, not the system.”

  • Logbook — the miss list, three lines: Matched: ___ · Missed: ___ (best guess why) · Change next: ___
  • System diff sentence: “I thought the system worked like ___; after building it, I discovered it worked like ___.”
  • Carry into the lab: open lab-024.html and reuse the corrected rule, not the reference sound.
