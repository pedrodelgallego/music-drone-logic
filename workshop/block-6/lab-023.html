Block 6 · Lab 023 · Track lab

Rule-Based Sequencing

You stop writing notes and start writing rules: a pool of allowed pitches, a probability table, a density limit. The rules play; you listen, adjust one number at a time, and curate the output into a piece. The music is ambient because the rules are gentle — the method is the whole block in miniature.

Track lab · S3 · D4 E2 M1 F2 · generative ambient · full cycle · new concepts: 5

listen → hypothesis → draw the path → calibrate → build → fault → diagnose → print → delta
Start here
  • Run the ten listening references below first; the hypothesis they produce is the start of the build.
  • Save a new set as LAB-023-rules.
  • This is an S3 lab: both specs are authored by you from the prompts below, on paper, before any gear.
  • Declare state, seed, and boundary before the first note — the Block 6 ritual, new in this lab and repeated in every lab of the block.
  • Expected time: one full cycle — two to three weeks of part-time sessions.
The point

Rules generate; you curate. The composer’s job moves one level up — not which note comes next, but which notes are allowed, and how often. A rule you cannot write down as a number is a wish, not a system.

How this lab works

Write the rules on paper

  • A pitch pool of five to seven allowed notes — nothing outside it may sound.
  • A probability table: per step or per beat, the chance that a note fires and the weight of each pitch.
  • A density rule: the maximum number of events per bar — silence is a rule too.

Let the system run

  • Live 12’s generative MIDI tools (or the Digitakt’s trig conditions) hold the rules; the MicroFreak is the voice.
  • Ten minutes unattended, hands off, recorded in full — the block’s self-running test, run here for the first time.
  • Adjust one number, run again — the one-variable rule applies to rules themselves.

Curate, don’t compose

  • Record the runs in full; choose 6–8 minutes from the hour.
  • Changing a rule between runs is allowed; grabbing an individual note is forbidden — that is the boundary.
  • Every musical feature in the final piece must be traceable to a rule you can name.
Pass standard
Minimum passAudible original piece, audio-path and control-path diagrams (two separate drawings), fault entry, print map, and system diff exist.
Solid passYou can point at any feature of the piece and name the rule that produced it — and the 10-minute unattended run neither died nor clipped.
Advanced transferA fresh run from the logged rule values produces statistically similar music, and you can predict what a rule change will do before you make it.

New concepts — in plain words

ConceptIn plain words
Audio path vs control pathTwo different rivers. The audio path carries sound — cables, tracks, buses, converters. The control path carries decisions — MIDI, rules, probabilities, mappings. In this block they are always drawn as two separate diagrams, because sound and decisions fail in different ways.
Generative ruleAn instruction that produces notes instead of being notes — a recipe instead of a meal. “Play one pitch from this pool every 1–4 beats, low notes twice as likely” is four bars of music you never wrote.
Probability tableA list of outcomes with a weight on each — the loaded dice of the system. 60% rest, 30% root, 10% fifth is a musical style expressed as three numbers.
Seed / repeatabilityWhere the randomness starts. A seeded system replays the same “random” choices on demand; an unseeded one never repeats. Most of your tools are unseeded — so the logged rule values, run number, and date are your seed substitute. If it isn’t written down, the run is gone.
CurationThe human role in a generative system: not playing, choosing. The system makes an hour; you decide which six minutes were music and answer for the decision.
audio path vs control path generative rule probability table seed curation

Listening — the hypothesis source

Ten works in which rules did the composing — tape arithmetic, counterpoint laws, probability tables, code, circuits. Each gets a brief: what to listen for before you press play, and how to check yourself after.

01 Brian Eno — “1/1” (Music for Airports) the rule is the arrangement

A handful of recorded piano phrases set on long cycles that drift past each other — the “arrangement” is whatever the cycles happen to produce, and it never produces the same minute twice.

Listen for
  • How few distinct phrases there actually are — count them on one pass.
  • Recurrence without exact repetition: the same phrase returns, but its neighbors are always different.
  • The absence of intention — no phrase answers another, yet your ear keeps inventing dialogue between them.
After — verify

How many phrases did you count? A small handful — the piece’s wealth comes from combination, not material. Did any pairing repeat exactly? Effectively never: the cycles are long enough that the music outlives your memory of it. If you heard “composition,” relisten tracking one phrase only and notice that its returns obey a cycle, not a feeling.

02 Brian Eno — Reflection rules at album scale

A generative system released both as software that runs forever and as an album — the album is one print of one run; the rules, not the notes, are the work.

Listen for
  • Changes of density and register arriving without sections — no event ever announces itself.
  • The boundary at work: nothing harsh, nothing fast, no register leaps — what has been forbidden is audible as a style.
  • Whether you can catch the system making a “choice” — a moment where you sense an alternative existed.
After — verify

What would a second print of this system share with the one you heard? The rules — the palette, the pace, the boundary — and nothing else; every note would differ and the piece would still be unmistakably itself. That is the target of this lab: identity living in the rules, not the output. If that seems abstract, relisten to any ten minutes and ask what stayed constant — it is never a note.

03 Laurie Spiegel — The Expanding Universe logic as counterpoint

Bell Labs’ GROOVE system running compositional logic in real time — algorithmic processes moving several voices at once, and sounding warmer than most hands do. Builds directly on Block 1’s “Patchwork” lesson; this is the same mind at longer breath.

Listen for
  • Several voices changing together in ways one pair of hands could not perform — the tell of a process.
  • Slow harmonic drift that feels inevitable rather than chosen — the rule is steering, not the moment.
  • Where the humanity lives: in the rule design and the curation, not in any played gesture.
After — verify

Could a performer have played this? The individual lines, perhaps — but not the coordinated drift of all of them; the coherence across voices is the algorithm’s signature. Where “Patchwork” showed you logic at pattern scale, this shows it at form scale. If you heard it as “played,” relisten following two voices simultaneously and notice they turn together, like a flock.

04 Lejaren Hiller & Leonard Isaacson — Illiac Suite, Experiment 1 the first computed counterpoint

1956: a computer generates a string quartet by applying the rules of species counterpoint — the first famous answer to “what happens if the rules compose alone?”

Listen for
  • Music that is audibly correct — no forbidden parallels, lawful cadences — yet strangely directionless.
  • The difference between legal and intended: every move is permitted; almost none feels wanted.
  • Moments where the rules accidentally produce beauty — and how you noticed them.
After — verify

What was missing? Not correctness — desire. The rules guaranteed grammar, never meaning, and the gap between the two is exactly the space your curation must fill in this lab. Relisten to one movement and mark the two or three passages that work as music; those are the moments a curator would have kept, and the computer could not tell them apart from the rest.

05 Terry Riley — “In C” 53 rules, human runtime

Fifty-three short modules and one page of instructions; every performer decides locally when to advance. The score is a generative system whose computer is the ensemble.

Listen for
  • The pulse as the only fixed clock — everything else is local decision inside a global rule.
  • Harmonies forming when modules collide — chords nobody wrote, produced by the overlap.
  • The ensemble drifting apart and re-clumping — how the simple “stay within two or three modules of each other” boundary holds the form together.
After — verify

Who decided the form you heard? Everyone and no one — each player made local choices, and the global arc emerged from the rule. A generative system does not need a computer; it needs a boundary and a clock. If the piece read as chaos, relisten following the pulse and notice how narrow the actual freedom is — that narrowness is the design.

06 David Behrman — “On the Other Ocean” the machine listens back

A 1977 home-built computer listens for specific pitches from flute and bassoon and answers with shifting synthesizer harmonies — a rule system with ears: “when you hear X, do Y.”

Listen for
  • Cause and effect: a played note, then the electronics tilting in response — find three clear instances.
  • The delay between trigger and answer — the system’s reaction time is part of its character.
  • How few rules produce something that feels like conversation.
After — verify

Who was following whom? Both — the players learn what the machine rewards and start playing to it, which means the rule changed the humans too. An interactive rule is still a rule; the input is just live. If you couldn’t hear the causality, relisten to the opening minutes where the texture is thin and each trigger is exposed.

07 Iannis Xenakis — “ST/10-1, 080262” probability as score

A composition computed on an IBM 7090 from probability distributions — density, pitch, duration, and instrument all drawn from weighted chance. A probability table performed by ten musicians.

Listen for
  • Texture as statistics: passages you would describe with a number — “sparser here, denser there” — before you would describe them with a melody.
  • Events that mean nothing alone and everything in mass — single plucks and scrapes that only add up at the scale of seconds.
  • Section boundaries marked by a change of distribution, not a change of theme.
After — verify

What did the composer actually choose? Not notes — distributions: how likely, how dense, how long, drawn fresh for each event. That is this lab’s probability table pushed to full scale. If the piece read as random noise, relisten comparing two sections as weather systems — one drizzles, one hails — and the form appears.

08 Mileece — Formations code-grown ambient

Generative ambient written as code — small sonic cells seeded and left to grow, the composer working as a gardener of processes rather than a writer of lines.

Listen for
  • Organic surface, procedural skeleton — the music sounds grown, not assembled.
  • Cells that recur transformed: the same material at new pitches, spacings, or speeds.
  • How the music breathes without a clock you can tap to.
After — verify

Did it sound like a machine? Mostly not — which is the finding: rule-based does not mean rigid, and gentleness can be a parameter. The “organic” feel is itself generated: irregular spacing, weighted choices, soft envelopes. Relisten and try to find a grid; failing to find one in code-made music tells you where the craft went.

09 Tristan Perich — 1-Bit Symphony the score is a circuit

A microchip in a CD case performs the code live every time you switch it on — there is no recording on the medium at all, only rules waiting to run, with one bit of output: on or off.

Listen for
  • Complexity built entirely from on/off — arpeggios and counterpoint with no timbre to hide behind.
  • The ingenuity forced by the constraint: how “loudness” and “space” get faked with rate and pattern.
  • That nothing you hear was ever audio until this playback — the piece exists only as logic.
After — verify

Where does this piece live? In the rules — the circuit is the score, and every playback is a fresh performance of them. This is the purest version of the lab’s claim: the work is the system. If the one-bit palette wore on you, relisten to one movement asking only what the rules must say — you will start hearing code, which is the point.

10 Marsen Jules — The Endless Change of Colour one source, endless run

A single short phrase fed to an algorithm that recombines it without end — the released album is an hour-long excerpt of a piece with no natural stopping point.

Listen for
  • How much variety one source yields when only the rule changes its view of it.
  • The absence of arrival: no minute is more “the piece” than any other.
  • Your own curation reflex — where would you have cut the excerpt, and why there?
After — verify

Where did the piece begin and end? Arbitrarily — someone chose a window onto a process that runs forever, which is precisely your final task in this lab: the system makes an hour, the curator answers for six minutes. If every minute sounded the same, relisten to the first and last five minutes back to back; the drift is real, just slower than attention.

Requirement

Your written hypothesis names where this piece’s rules will live and what they may decide — cycles drifting past each other, a boundary audible as style, logic moving voices together, lawful-but-blind counterpoint, modules with human runtime, a rule with ears, probability as weather, gentleness as a parameter, the work existing only as logic, or a window cut from an endless run — and what was stolen from the comparison. 1 · The two specs — author from prompts

This is an S3 lab: no fields are provided. Answer each prompt in writing — your answers are the spec — then correct every row against reality after the build.

System Spec — author from these prompts
InputsWhich voice plays the rules — MicroFreak preset, Live instrument? Write the pitch pool as note names: five to seven, no more. What plays besides the generator, if anything?
TransformsWhich rules sit between the pool and the voice — probability per step, density limit, octave rule, rest weight? Write each as a number you could re-enter from the page.
Feedback / variationWhat keeps an hour alive — a slow modulation on the rule values themselves, a second rule layer, or nothing but the combinatorics? Where does variation come from when no one is playing?
ConstraintsTrack budget? One-variable rule between runs? What is forbidden to the system — this row feeds the boundary declaration below.
Human roleWhere exactly do you act — rule design, run selection, the final cut? Name what you may not do (touch individual notes) and where that line is written down.
Engineering Spec — author from these prompts, correct after building
Signal pathDraw both: source → recorder and source → ears. Which M4 input carries the MicroFreak? Where does the generated MIDI travel — and does it appear on your control-path diagram, not the audio one?
Gain planA generative system can fire any density it likes — what peak level do you allow for the densest legal bar, and where is it set? Quote the lab-001 ledger value you are using.
Clock / syncWho is master clock — Live? Does the generator follow the transport or free-run? What happens to the rules when the transport stops and restarts?
LatencyIs anything monitored live while the system runs? If the run is recorded unattended, what does latency still affect — and what can you safely ignore?
Stereo / phaseIs the voice mono? If width is added, where — and does the piece survive a mono check?
Noise budgetLong quiet stretches are legal output — what will the noise floor be doing during them? Record thirty seconds of “silence” first and write down what you hear.
Print strategyDo you print every run in full, or only candidates? When a run cannot be reproduced (see seed), what does that force you to print, and when?
Failure modePredict one likely failure and how you would see it. This lab’s faultType is seed — the canonical failure is a beautiful passage no logged value can regenerate.
SafetyAn unattended system plays to an empty chair — set a level ceiling before you leave the room, and write the phones level you will return to.

2 · State · seed · boundary — declare before the first note

The Block 6 ritual: every lab in this block files this declaration alongside the specs. One honest sentence per row beats a paragraph of hope.

Decision-system declaration — authored, then corrected after the build
StateWhat does the system remember? Where do the rules physically live — Live set, device preset, paper? What survives closing the set, and what evaporates?
SeedCan a run be reproduced? If your tools expose no seed, declare the substitute: rule values, run number, date, and the printed audio itself. Anything not logged is declared lost in advance.
BoundaryWhat may the system decide — pitch from the pool, event placement, density up to the limit? What is forbidden — pitch outside the pool, structure, the final cut? The forbidden list is the composition.

3 · Bench preflight — two paths, drawn separately

Block 6 draws every system twice. The audio path carries sound; the control path carries decisions. They share no arrows — if one drawing needs an arrow from the other, you have found a place where a decision becomes audible, and that joint is where faults live.

Audio path — what carries sound

MicroFreak out → TRS → M4 in 1 → USB-C → Live audio track → master → M4 → DT 990

Control path — what carries decisions

rule sheet (paper) → entered → Live 12 MIDI generators / transformers → USB MIDI → MicroFreak voice
  • Honest rig note: the Digitakt II’s trig conditions, chance, and fill logic are a hardware home for the same rules — and the gear roadmap’s Torso T-1 is this lab’s dedicated path when it arrives. The rules transfer; only the menus change. Whatever holds the rules, it appears on the control diagram only.
  • 10-minute unattended test: this lab self-runs, so it takes the block’s test — start the system, hands off for ten minutes, record everything. Pass: it neither falls silent past its own rules nor clips the densest bar. Fail: fix a rule or a boundary, never a moment.
Pre-flight — before the first note
  • Both specs authored on paper from the prompts — this is S3: blank pages, your answers
  • State, seed, and boundary declared — one sentence each, filed with the specs
  • Both diagrams drawn — audio path and control path, two separate drawings
  • Gain ledger from lab 001 applied at M4 in 1; densest-legal-bar peak level chosen and set
  • Control path verified silently first: rules emit MIDI, meters move, no audio yet
  • Every rule value written as a number in the logbook — “some probability” is not a measurement
  • Phones at the lab-001 mark; level ceiling set before any unattended run

5 · Diagnosis & report

Fault ritual — faultType: seed

Predict the symptom → create the fault → observe → diagnose → fix → write the rule. The fault: close the set, reopen it fresh, and attempt to regenerate yesterday’s best passage from the logged rule values alone. Where the regenerated music differs in kind (not just in notes), a value went unlogged — find which one, log it, and write the prevention rule. The expected lesson: an unseeded system is reproducible only as a class, never as a take — which is why the take gets printed.

The full report bundle — ten items, filed in the logbook
  • Rendered piece — 24-bit WAV, master peaking −3 dBFS, 6–8 minutes, curated from the runs
  • Routing diagrams — audio path and control path, drawn before patching, corrected after; the diff is the lesson
  • Recording plan — which runs were printed, in full or in part, and when
  • Patch notes — every rule value as a number: pool, weights, density limit, run log
  • Session notes — per run: what the rules did that you did not expect
  • Reflection answers, including the listening comparison — which reference contributed what to the hypothesis
  • One removal decision — a rule or a passage cut, and what cutting it did
  • Fault entry — symptom, cause, fix, prevention rule (faultType: seed)
  • Print map — which runs exist as audio, which rule sets as text, and which passages are gone forever
  • System diff sentence — “I thought the system worked like X; after building it, I discovered it worked like Y.”
  • Minimum pass gate: audible output · routing diagrams (both paths) · fault entry · print map · system diff. Without these five, the lab is incomplete regardless of audio quality — a musically strong piece cannot bypass the engineering spine.
Track-lab principle

The rules generate, the boundary protects, the human curates — and nothing the system makes is yours until you can name the rule that made it and the log that could make it again.
